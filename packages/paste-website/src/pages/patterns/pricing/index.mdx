## About

The Pricing pattern should be used when the user is presented with pricing or packaged options to select from. Use this pattern to clearly show the differences between different plans in order for the user to make an informed decision.

Examples:
<UnorderedList>
  <ListItem>Selecting an account plan on upgrade.</ListItem>
  <ListItem>Switching between a free and paid feature.</ListItem>
  <ListItem>Adding a packaged plan to an account based on a use case.</ListItem>
</UnorderedList>

The information presented in the layout will be dependent on the packaged plans themselves.

## General

Pricing layouts should adhere to the following guidelines:
<UnorderedList>
  <ListItem>Limit the number of pricing Cards to 4, with a minimum of 2 as to not overwhelm users with options. For a larger quantity of plans, consider reducing the options or using a <Anchor href="/components/table">Table</Anchor>.</ListItem>
  <ListItem>Display the current plan (if any).</ListItem>
  <ListItem>Use a primary Button and Badge for the recommended option (if any), all other pricing cards use a secondary Button for selection.</ListItem>
  <ListItem>
    Limit the listed features to 8 items, focusing on the differences between each package
    <UnorderedList>
      <ListItem>If the lower packages are included in the higher end packages, we recommend the first line item of the next package mention the previous plan. Example: “All features from Basic plan”.</ListItem>
    </UnorderedList>
  </ListItem>
  <ListItem>Always order the cards left to right from least expensive to most expensive, or least number/value of features to most number/value of features. The current selection remains in this order (it does not move to the beginning of the list).</ListItem>
</UnorderedList>

## Variations

### Cards

Use the card layout to display the user’s options on pages that may redirect to a [Wizard flow](/page-templates/wizard) or when this selection is the only step (besides a confirmation) the user needs to take to change their plan.

The buttons on these cards can redirect to a billing flow. The card also allows for more detail about the plan.

Use a Visual Picker when the selection is happening in a wizard and the user will be continuing on to additional steps besides a simple confirmation (e.g., payment information).

The Visual Picker should be more succinct in the detail it shares.

## Responsiveness

For larger viewports, cards and pickers should be side-by-side. For smaller viewports, stack the cards or picker vertically.

For cards, the height of the cards should be equal to each other even if the number of features, badges, or buttons are different. You can use CSS Grid to ensure the cards are the same height.

<Callout variant="neutral" marginY="space70">
  <CalloutText>
    Tip: If the total height of the cards is too long for small viewports, consider using a <Anchor href="/components/summary-detail">Summary Detail</Anchor> to collapse the package features, displaying them only when the user interacts with it.
  </CalloutText>
</Callout>

## Figma

By default, the Figma component is built with predefined minimum and maximum widths.

The auto layout settings ensure that the number of features, badges, avatars, or buttons don’t affect the card height, and the "More info" button stays anchored at the bottom.
In order to maintain proper space in between card elements you may need to manually adjust the card group container height.

## Do’s and Don’ts

<DoDont>
  <Do title="Do" body="Only include the additional features in the list of each plan." />
  <Dont title="Do" body="Don’t show the negative or missing features of each product, or change the icon to an “X”." />

  <Do title="Do" body="Include the “free” or “included” plan even if it's default on the account." />
  <Dont title="Do" body="Don’t only have 1 pricing Card on a page. Users should see all of their options." />

  <Do title="Do" body="Keep the content of the feature list succinct and jargon-free; use the “more info” option for more info." />
  <Dont title="Do" body="Don’t put a large amount of content in a feature item, and don’t put more than 8 features" />
</DoDont>
